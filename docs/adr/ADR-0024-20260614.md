# ADR-0024: Use Redis as Primary Cache Layer Instead of Memcached

**Status:** Accepted
**Date:** 2026-06-14
**Author:** P-r-e-m-i-u-m
**Reviewers:** Engineering Team

---

## Context

Our application requires a high-performance caching layer to reduce database load. We evaluated multiple in-memory caching solutions after observing that 70% of our database queries were repetitive reads. At peak load (1000 req/s), our PostgreSQL instance was hitting 80% CPU utilization.

## Decision

We will use Redis 7.x as our primary caching layer with a two-level cache strategy: in-process LRU cache for hottest data backed by Redis for distributed caching across instances.

## Alternatives Considered

| Option | Pros | Cons | Decision |
|-|-|-|-|
| Chosen approach | Best fit for requirements | Some tradeoffs | Accepted |
| Alternative A | Simpler | Missing key features | Rejected |
| Alternative B | More features | Too complex / costly | Rejected |

## Consequences

### Positive
Database CPU reduced from 80% to 22% at peak load. p99 latency on cached endpoints: 1.2ms (was 45ms). Cache hit ratio: 94% after warm-up period.

### Negative
Redis becomes a critical dependency and requires HA setup in production.
